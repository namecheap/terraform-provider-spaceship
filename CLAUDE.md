## Commands

```bash
go build .                                              # Build
make test                                               # Unit tests (excludes acceptance)
go test -run TestFunctionName ./internal/provider       # Single unit test
make lint                                               # Lint (linters + formatter check)
make fmt                                                # Auto-fix formatting
make docs                                               # Generate docs
make docs-validate                                      # Validate docs match schema

# Acceptance tests — only run when explicitly asked
# Requires: SPACESHIP_API_KEY, SPACESHIP_API_SECRET, SPACESHIP_TEST_DOMAIN
make testacc
go test -run TestAccFunctionName ./internal/provider -v # Single acceptance test
```

## Verification workflow

After making changes, follow this order:

1. **Unit tests** — `make test` (or `go test -run TestName ./internal/provider` for a specific test)
2. **Lint** — `make lint` (fails on lint rules or unformatted files; run `make fmt` to auto-fix formatting)
3. **Build** — `go build .`
4. **Acceptance tests** — run only when the user explicitly asks. These hit real APIs and modify real DNS records.

## Architecture

[Terraform Plugin Framework v1](https://github.com/hashicorp/terraform-plugin-framework) provider (protocol 6) for managing Spaceship domains and DNS.

- **All Spaceship API access goes through the external SDK** [`github.com/namecheap/go-spaceship-sdk`](https://github.com/namecheap/go-spaceship-sdk). The provider imports `.../client` (HTTP client, domains, DNS records, personal nameservers, record matching) and `.../client/records` (per-type wire structs + `Validate*` methods). There is no longer a bundled client in this repo. For anything about how an API call behaves — endpoints, serialization, pagination, error mapping, rate-limit fallback, record matching, per-type validation rules and their verified API quirks — **the SDK is the source of truth**; read its code and its `CLAUDE.md`.
- This repo owns only Terraform-specific concerns: schemas, state/diff reconciliation, import lifecycle, provider config, and the `validator.Object` wrappers in `internal/provider/records/` that adapt the SDK's record `Validate*` methods to Terraform.
- `docs/` is autogenerated — run `make docs` to regenerate. Do not put custom content there.
- Internal design notes live in `internal/docs/`.
- Provider reads `SPACESHIP_API_KEY` / `SPACESHIP_API_SECRET` env vars or inline HCL attributes.

## Testing strategy

This provider uses a **two-layer** testing approach. Do not duplicate coverage between layers.

- **Provider-layer unit tests** (`internal/provider/`) — Test Terraform-specific concerns: schema validation with Terraform types, state management, import lifecycle, provider configuration, and diff/reconciliation logic. Do **not** re-test SDK logic (e.g., don't spin up an httptest server just to verify that `if err != nil` adds a diagnostic, and don't re-assert API serialization/pagination/validation — those are the SDK's tests).
- **Acceptance tests** (`*_acc_test.go`) — Test end-to-end Terraform lifecycle (plan/apply/destroy/import) against the real Spaceship API. These are the authoritative tests for "does it work."

Client-layer unit tests (API calls, HTTP serialization, pagination, error mapping, record-validation logic) now live in the **SDK repo**, not here.

When adding tests, ask: "Is this testing something unique to this layer?" If the answer is no, the test belongs elsewhere or doesn't need to exist. See `internal/docs/testing.md` for detailed guidelines.

## Key design rules

These behaviors are implemented in the SDK and the provider relies on them; if you change provider diff/reconciliation logic it must stay aligned. See the SDK's `CLAUDE.md` for the authoritative descriptions.

- **DNS records are scoped to the custom group**: The API returns records across three DNS groups: `custom`, `product`, and `personalNS`. Only `custom` group records are managed. Do not touch records in other groups.
- **DNS record matching mirrors the API**: Records are matched by type + name + data using case-insensitive comparison, **except** TXT values which are case-sensitive. The provider's diff logic must follow the same rules the SDK does.
- **Rate-limit fallback**: The SDK's domain info call falls back to the domain list API on HTTP 429. Don't add provider-side logic that defeats this.

## Adding a DNS record type validator

The wire struct and its `Validate*` methods live in the **SDK** (`client/records/<type>.go`); this repo only adds the Terraform `validator.Object` wrapper and registers it. Copy an existing type (A, AAAA, ALIAS, CAA, SRV) rather than inventing a shape:

1. **SDK: struct + `Validate*` methods** in the SDK's `client/records/<type>.go` (with unit tests). Mirror API-documented constraints only — do not invent stricter rules. Then cut/`go get` an SDK release so the new type is importable here. See the SDK's `CLAUDE.md`.
2. **Provider `validator.Object`** in `internal/provider/records/record_<type>_validator.go`, calling the SDK struct's `Validate*` (imported as `clientrecords`). The validator runs against every record in the list and must short-circuit when `type != "<TYPE>"`. Only validate type-specific fields — `name` and `ttl` already have schema-level validators on the attribute.
3. **Register the validator** in the `Validators` slice on the `records` nested block in `internal/provider/dns_records_resource.go`. Nothing enforces that every supported type has a registration — a missing one silently skips validation.

Acc tests (`record_<type>_validator_acc_test.go`) own API edge cases: unusual values, update/delete+upsert paths, case-sensitivity round-trips, same-host multiplicity. The `plancheck.ExpectEmptyPlan()` on a re-apply step is the strongest cheap signal that the full config → write → read → state cycle converges.

## Release process

Releases are semi-manual and maintainer-gated: merges to `master` accumulate into a release-please "Release PR"; a maintainer merges that PR to cut a tag, and GoReleaser publishes the binary. See [RELEASE.md](RELEASE.md) for the full flow, including the manual Terraform Registry resync that is occasionally required.

**Only releasing commit types cut a release**: `fix:` → patch, `feat:` → minor, `feat!:`/`BREAKING CHANGE:` → major; `chore:`/`ci:`/`refactor:`/`test:`/`docs:` are non-releasing. The trap is **published artifacts** — anything shipped to users must ride a release:

- **Runtime deps** (`go.mod`) ship in the binary → commit as `fix(deps):`, never `chore(deps):` (which never releases, so the bump reaches nobody).
- **Registry docs** (the `docs/` tree) are ingested by the Terraform Registry only at a release tag → correct them with `fix(docs):`. Repo-internal markdown (README, `RELEASE.md`, `CLAUDE.md`) stays `docs:`.

For Dependabot this is enforced via `commit-message.prefix` in `.github/dependabot.yml` (gomod → `fix`, github-actions → `ci`).
